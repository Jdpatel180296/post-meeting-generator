## ğŸ“ Architecture Highlights

### Why This Design?

**Multi-Page App (React Router)**

- Different screens for different tasks
- Users don't need all features at once
- Cleaner UI, better UX
- Easier to extend with new pages

**Backend API**

- Separation of concerns
- Can be called from mobile/desktop apps later
- Proper HTTP semantics
- Scalable to multiple servers

**Database Layer**

- PostgreSQL for reliability
- Migrations for version control
- Proper relationships (foreign keys)
- Indexed queries for performance

**AI Integration**

- OpenAI GPT-4 for quality content
- Platform-specific prompts (LinkedIn vs Facebook)
- Custom prompts via automations
- Error handling for API limits

## âœ¨ Key Technical Decisions

### 1. Inline CSS vs CSS Framework

**Decision:** Inline CSS with React style props
**Why:** No external dependencies, self-contained components, easy to modify

### 2. React Router v6 vs v5

**Decision:** React Router v6
**Why:** Latest features, hooks support, cleaner API

### 3. Database: Knex vs Raw SQL

**Decision:** Knex query builder
**Why:** Parameterized queries (safe), migrations, database abstraction

### 4. AI: OpenAI vs Other LLMs

**Decision:** OpenAI GPT-4
**Why:** Most reliable, good API, widely used, good documentation

### 5. State Management: useState vs Redux

**Decision:** useState (React Hooks)
**Why:** Sufficient for current complexity, no over-engineering

## ğŸ“ Support & Debugging

### Common Issues & Solutions

**Issue: Server won't start**

- Check PostgreSQL is running: `brew services list | grep postgres`
- Check port 4000 not in use: `lsof -i :4000`

**Issue: Client shows blank page**

- Check server running on 4000
- Check network tab in DevTools
- Clear browser cache

**Issue: Database connection failed**

- Verify PostgreSQL running
- Check .env has correct credentials
- Verify postmeeting database exists

**Issue: OpenAI not generating content**

- Check OPENAI_API_KEY is set
- Verify key is valid at platform.openai.com
- Check API account has credits
